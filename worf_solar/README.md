# Solar & Battery — วิเคราะห์เพื่อการขาย (คู่มือภาษาคน)

เว็บแอปเล็กๆ (Streamlit) สำหรับทีมขาย: โยนไฟล์ 2 อัน แล้วระบบคัดแพ็กเกจที่เหมาะกับ
ลูกค้า พร้อมตัวเลขการเงินและไฟล์สรุปหน้าเดียวส่งลูกค้าได้เลย

## ใช้งานยังไง (ง่ายมาก)

1. เปิดเว็บ ใส่รหัสผ่าน
2. ที่ **แถบซ้าย** อัปโหลด 2 ไฟล์:
   - ไฟล์บิล/พฤติกรรมลูกค้า (รองรับทั้งราย 15 นาที และบิลรายเดือน ตามเทมเพลตไทย)
   - ไฟล์ catalog แพ็กเกจสินค้า (ไฟล์หลังบ้าน)
3. ตั้งค่าที่แถบซ้ายครั้งเดียว: ลูกค้าเอาแบตไหม, ช่วงความจุแบต, ระบบไฟ, สมมติฐานการเงิน
4. ระบบแสดงผลอัตโนมัติ: สรุปการใช้ไฟลูกค้า → แพ็กเกจแนะนำ (คัดมาให้เหมาะ ~10 อันดับ)
   → ตารางการเงิน → กราฟจุดคืนทุน → ปุ่มดาวน์โหลดสรุปหน้าเดียว

**ไม่ต้องอัปไฟล์ซ้ำหลายที่** — อัปครั้งเดียวที่แถบซ้าย ใช้ได้ทั้งหน้า

## ความปลอดภัยข้อมูล

ระบบซ่อนราคาต้นทุน (Pro price) และ margin% หลังบ้านอัตโนมัติ โชว์ลูกค้าเฉพาะ
ราคาลงทุน (Catalog price) เท่านั้น

## ไฟล์ในโปรเจค

| ไฟล์ | หน้าที่ |
|---|---|
| `app.py` | หน้าเว็บทั้งหมด |
| `data_parser.py` | อ่านไฟล์บิลลูกค้า (เทมเพลตไทย ราย 15 นาที/รายเดือน, รองรับ xlsx/xls/csv) |
| `packages.py` | อ่านไฟล์ catalog + ซ่อนต้นทุน/มาร์จิน + คัดกรอง/จัดอันดับ (`recommend_packages`) |
| `finance.py` | สูตรการเงิน: คืนทุน / NPV / IRR |
| `sim_bridge.py` | เชื่อมข้อมูลลูกค้าจริง → simulator เพื่อคำนวณผลประหยัดต่อแพ็ก |
| `battery.py`, `tariff.py`, `optimizer.py`, `simulator.py` | เครื่องมือจำลองระบบ (ตอนนี้ถูกเรียกใช้จริงผ่าน `sim_bridge.py` แล้ว) |

## ติดตั้งและรัน

```
py -m venv venv
.\venv\Scripts\Activate.ps1
python -m pip install -r requirements.txt
python -m streamlit run app.py
```

**ต้องตั้งรหัสผ่านก่อนรันเสมอ** สร้างไฟล์ `.streamlit/secrets.toml` ใส่:
```
APP_PASSWORD = "รหัสที่คาดเดายากจริงๆ"
```
หรือตั้ง environment variable `APP_PASSWORD` ก็ได้ ระบบจะ**ปฏิเสธการรัน**ถ้าไม่ตั้ง
หรือตั้งเป็น `changeme` (กันลืมเปลี่ยนรหัสตอน deploy) และมี lockout + session timeout ในตัว

## สรุปการแก้ไขในเวอร์ชันนี้ (v14)

- **กราฟพฤติกรรมใช้ขนาดเดียวกับ "3 ตัวเลือกที่แนะนำ"** — เดิมกราฟโชว์ขนาดจากแคตตาล็อก
  (5/30/51 kWp) แต่ตัวเลือกที่แนะนำเป็นคนละขนาด (41/102/163 kWp) ทำให้ไม่สอดคล้องกัน
  ตอนนี้ย้ายการคำนวณ quant ไปก่อนวาดกราฟ แล้วกราฟใช้ขนาด ขั้นต่ำ/คุ้มสุด/ใกล้เป้าหมาย
  ชุดเดียวกัน (แสดงจำนวนชุดด้วย) · เส้น SOC แบตอ้างอิงแบตของตัวเลือก "คุ้มสุด"
- **เพิ่มคอลัมน์ยี่ห้อ** ในตาราง 3 ตัวเลือก + การ์ด + ไฟล์สรุปส่งลูกค้า (อินเวอร์เตอร์/แบตเตอรี่/รหัสแพ็ก)
- **ผลผลิตโซลาร์ดีฟอลต์ 4.0 → 4.5** kWh/kWp/วัน
- **เอาแถบ "สมมติฐานการเงิน" ออก** (อายุโครงการ/ค่าไฟขึ้น/อัตราคิดลด/จำนวนแพ็ก)
  เปลี่ยนเป็นค่าคงที่ในโค้ด (10 ปี / 3% / 5% / 10 แพ็ก) แก้ที่ตัวแปร HORIZON_YEARS ฯลฯ ได้
- **เพิ่มไฟล์ตัวอย่างสำหรับโหมดสแกนลูกค้า** (.zip มี 3 ราย: โรงงานกลางวัน → เกรด A,
  ร้านค้าเปิดถึงค่ำ → เกรด B, โรงงานกะกลางคืน → เกรด C) ลากใส่ทดสอบได้ทันที

**แก้บั๊กสำคัญ 2 จุดที่เจอตอนทดสอบ:**
- *"มีข้อมูลเพียง 0 เดือน"*: ฟังก์ชันประเมินแนวโน้มตัดเดือนที่มี <20 วันทิ้ง ทำให้ไฟล์
  ตัวอย่าง (1 วัน/เดือน) เหลือ 0 เดือน → เปลี่ยนเป็น normalize เป็นฐาน 30 วันแทนการทิ้ง
- *แนวโน้มหลอก −9.7%/ปี*: เดิมใช้ความชันเส้นตรงกับข้อมูล 12 เดือน ซึ่ง**จับผลฤดูกาล**
  (ไทยร้อนสุด เม.ย. หนาวสุด ธ.ค.) มาเป็น "แนวโน้มลดลง" ทั้งที่การใช้ไฟไม่ได้ลดจริง
  → ตอนนี้ใช้ **YoY จับคู่เดือนเดียวกันคนละปี** (≥13 เดือน) หรือ 12 เดือนล่าสุด vs
  12 เดือนก่อน (≥24 เดือน) · ถ้ามี ≤12 เดือน จะ **ไม่เดา** ใช้ 0%/ปี พร้อมบอกเหตุผล

## สรุปการแก้ไขเวอร์ชันก่อนหน้า (v13 — Quant)

โมดูลใหม่ `quant.py` เพิ่ม 4 ความสามารถ (ทั้งหมดใช้ข้อมูลเดิม ไม่ต้องถามลูกค้าเพิ่ม):

1. **ประเมินแนวโน้มการใช้ไฟจากข้อมูลลูกค้าเอง** (แทนการถามว่า "จะขยายกิจการไหม" ซึ่งลูกค้ามักตอบไม่ได้)
   - ≥24 เดือน → เทียบปีต่อปี (ตัดผลฤดูกาลออก) = เชื่อมั่นสูง
   - 12-23 เดือน → ความชันแนวโน้ม = ปานกลาง
   - 3-11 เดือน → ความชัน แต่จำกัด ±5% เพราะฤดูกาลหลอกตา = ต่ำ
   - <3 เดือน → โต 0% + เตือนให้ขอข้อมูลเพิ่ม
   แล้วฉายภาพไปข้างหน้า 2-3 ปี เพื่อเผื่อขนาดระบบ
2. **Size sweep** — ไล่คำนวณ **ทุกขนาดที่ซื้อได้จริง รวมการติดหลายชุดขนานกัน** (สูงสุด 4 ชุด)
   บนโหลดจริงของลูกค้า แล้วเทียบ NPV/IRR/คืนทุน เพื่อหา "ขนาดที่คุ้มสุด"
   → พิสูจน์แล้วว่า **ขนาดใกล้โหลดสุด ≠ คุ้มสุด** (เช่น 117 kWp คืนทุน 5.0 ปี NPV 8.3M
   ดีกว่า 200 kWp ที่คืนทุน 6.8 ปี NPV 7.9M เพราะตัวใหญ่ผลิตไฟเหลือทิ้ง 43%)
3. **Monte Carlo** — สุ่มสมมติฐาน 300 รอบ (แดด ±10%, performance ratio 78-88%,
   ค่าไฟขึ้นเร็ว/ช้า, แผงเสื่อม 0.4-0.8%/ปี, **ค่าดูแลรักษา 0.5-1.5%/ปี ซึ่งเดิมไม่เคยคิด**)
   → เสนอลูกค้าเป็น **ช่วง P10-P90** แทนตัวเลขเป๊ะที่โดนแย้งง่าย + บอกโอกาสคืนทุนใน 5/7/10 ปี
4. **Lead Scoring** — ให้คะแนน 0-100 + เกรด A/B/C ว่าลูกค้ารายไหนน่าจะคุ้มโซลาร์
   (ใช้ไฟกลางวัน 40 · ขนาดการใช้ไฟ 25 · ความสม่ำเสมอ 20 · ใช้ไฟทุกวัน 10 · แนวโน้มโต 5)
   มีโหมด **สแกนหลายรายพร้อมกัน** อัปโหลดไฟล์มิเตอร์หลายไฟล์ → ได้ลิสต์เรียง A→C
   ดาวน์โหลด CSV ให้ทีมขายยิงตรงกลุ่มที่ปิดดีลง่ายก่อน

**ขนาดขั้นต่ำที่สมเหตุสมผล** (ตอบโจทย์: ทำไมไม่เสนอ 3 kWp ให้ลูกค้าใช้ 120 kWp):
คำนวณจาก **โหลดฐานตอนกลางวัน** (P25 ของช่วง 9-15น.) = ขนาดที่ผลิตแล้วใช้เองหมด
ไม่เหลือทิ้ง = คืนทุนเร็วสุดต่อบาทที่ลงทุน ไม่ใช่ "แพ็กเล็กสุดที่มีในสต็อก"

  แบบเดียวกับไฟล์วิเคราะห์ของทีม คำนวณจากข้อมูลจริงอัตโนมัติ + ตาราง Demand Peak รายเดือน
- **ตารางแพ็กเกจโชว์ 2 มุมคู่กัน**: "จากข้อมูลลูกค้า" (จำลอง/ประมาณการ) vs "แคตตาล็อก"
  ทั้งผลประหยัด/เดือนและระยะคืนทุน — ให้เซลชี้ให้ลูกค้าเห็นภาพจากพฤติกรรมตัวเอง
- **โชว์ยี่ห้อ+รุ่นแบตเตอรี่เสมอ** เมื่อลูกค้าเลือก "มีแบตเตอรี่" (แก้บั๊กชื่อแบรนด์แบตไม่ขึ้น)
- **สไลเดอร์เป็นสีดำ** (แก้ theme primaryColor จากม่วงอ่อน + CSS กันเหนียวทุกส่วน)
- **อัตราค่าไฟทศนิยม 4 หลัก** ดีฟอลต์ On-Peak 5.7982 / Off-Peak 2.6369 / **Ft 0.1623**
- ตัวกรองยี่ห้ออินเวอร์เตอร์ (เจาะ 5-10 ตัวเลือกต่อแบรนด์) · ปุ่มดาวน์โหลดเทมเพลตในแอป
- แก้บั๊กอ่านวันที่ไทยแบบ วัน/เดือน/ปี ที่เคยเสี่ยงสลับวัน-เดือนเงียบๆ (dayfirst)
